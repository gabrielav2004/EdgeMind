# EdgeMind

A lightweight semantic retrieval system for edge devices and robotics. No cloud, no vector database, no heavy runtime dependencies.

## The Problem

Semantic search typically requires vector databases, float32 embeddings, and cloud-hosted models — too heavy for edge hardware. EdgeMind replaces all of that with binary embeddings stored in a flat binary file, retrieved via two-stage hamming + float reranking, with responses generated by a fully local quantized model.

## How It Works

**Ingestion (runs once):**
```
documents → chunk → embed → binary quantize → flat binary file
```

**Retrieval + Response (runs on edge):**
```
query → binary embed → hamming search (top 15) → float rerank (top 3) → local model → answer
```

Two-stage retrieval: hamming distance finds candidates fast, float cosine reranks precisely. The response model runs fully local via llama.cpp — no internet required.

## Why Binary Embeddings

A 384-dimensional float32 embedding is 1536 bytes. Binary quantization converts each dimension to a single bit based on sign — the same embedding becomes 48 bytes. Retrieval uses hamming distance instead of cosine similarity, which is a native CPU bit operation.

Results from testing:
- 32x storage reduction vs float32
- 100% top-1 retrieval accuracy on domain-specific corpora
- Sub-millisecond hamming search at thousands of chunks
- Float reranking fixes rank-2 answer problem
- No vector database required

## Project Structure

```
EdgeMind/
  parse.py       # ingest txt, pdf, json into chunks
  store.py       # embed, quantize, write to binary database
  search.py      # two-stage hamming + float rerank retrieval
  respond.py     # local response generation via llama.cpp
  run.py         # unified CLI
  quantize.py    # verify binary embedding quality
  data/
    knowledge.bin  # flat binary database
    knowledge.idx  # byte offset index
    docs/          # place documents here
  models/
    tinyllama.gguf # local quantized model
```


## Retrieval Architecture

**Stage 1 — Binary hamming search**
Query is quantized to binary and XORed against all stored vectors. Top 15 candidates retrieved by minimum hamming distance. Extremely fast — pure bit operations.

**Stage 2 — Float rerank**
Top 15 candidates re-scored using full float32 cosine similarity. Final top 3 returned. Fixes rank-2 answer problem where binary search finds the right document but ranks it second.

## Database Format

`knowledge.bin` — one entry per chunk:
```
[48 bytes binary vector][2 bytes chunk length][raw utf-8 bytes]
```

`knowledge.idx` — one entry per chunk:
```
[8 bytes uint64 offset into knowledge.bin]
```

No third-party database. Two flat files. Read and written with Python's built-in `struct` module.

## Chunking

Documents are split at natural boundaries in priority order: paragraph break → sentence end → clause boundary → word boundary. Never splits mid-word.

```python
CHUNK_SIZE = 300    # characters per chunk
CHUNK_OVERLAP = 50  # overlap between chunks
```

## Performance

| Metric | Value |
|--------|-------|
| Storage per chunk | ~50 bytes + text |
| Hamming search (1k chunks) | < 5ms |
| Float rerank (15 candidates) | ~20ms |
| Total query latency | ~50ms |
| Embedding model size | 22MB |
| Response model size | 670MB (Q4) |
| RAM at runtime | < 1.5GB |

## Supported Input Formats

- `.txt` — plain text
- `.pdf` — via pypdf
- `.json` — recursively flattened to text

## Design Principle

Everything heavy runs at ingestion time on a capable machine. Everything that runs on the edge device is minimal. The knowledge base is two files. Retrieval is arithmetic. The model is local.
